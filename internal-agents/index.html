Message routing

From @mention to running code

Slack @mention → Turret webhook → routing.yaml → Persona loaded → Sandbox boots → Agent works → Reply in thread
  • @mention catch-all: any @bot in any non-dedicated channel dispatches to that bot — no config needed.
  • Per-channel bindings: dedicated channels hard-wired to specific agents
  • DMs: route to whichever bot you messaged directly
  • Proactive channels: agents post unprompted messages - e.g. #jack-feedback / #jill-feedback / #general-agent

Key insight: if you @jesse in any channel, Jesse’s sandbox boots. The first message takes around 60s, subsequent loads are faster.

Agent anatomy

Simple files define an agent

.agents/agents/jesse/
  AGENTS.md ← identity, model, env vars
  SOUL.md ← voice and style
  MEMORY.md ← cross-session context (auto)
  BOOTSTRAP.md ← only if unconfigured
  • AGENTS.md — the manifest: model, credentials, responsibilities
  • SOUL.md — personality: tone, verbosity, constraints
  • MEMORY.md — accumulates context across sessions automatically

All checked into git. Change a persona with a PR. No deploy needed — Turret reads the latest on next boot.

Expectations

What works well vs. what doesn’t

Works well

  • Defined, repeatable tasks with clear success criteria
  • Slack-native async workflows (questions, requests)
  • “Send this email”, “Look up X in Datadog”, “Summarize this PR”

Doesn’t work (yet)

  • Multi-hour workflows (sandbox lifetime limits)
  • Ambiguous judgment calls (blocks or guesses wrong)
  • E2E Browser flow testing (no credentials in  sandbox)
  • Write production DB access (all DB access is forked for safety)

When things go wrong

Transparent failure — you can always see why

No reply failure — sandbox didn’t boot, visible in dashboard

  • No reply? Sandbox failed to boot — check Turret logs on dashboard
  • Empty turns? Credential issue — Turnstile rejected the sentinel key
  • Wrong answer? Check what the agent was thinking.
  • Infinite loop? Agent retrying a failing command — kill the thread and try again.

Every failure is debuggable. The thread transcript shows every command that ran.
